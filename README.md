# Octopus 智能体系统

`octopus` 是一个基于看板驱动的多智能体协同系统，支持任务的自动分配、子 Agent 的动态孵化与释放、以及周期性任务调度。

## 核心特性

### 1. 弹性生命周期管理 (Auto-Scaling)

- **零任务自动释放**：看板区域任务完成后自动销毁子 Agent 实例
- **按需自动孵化**：新任务进入时动态启动新的子 Agent 实例
- **主 Agent 常驻**：作为系统核心守护进程持续监控调度

### 2. 看板驱动的任务流 (Kanban-Driven Workflow)

- **区域隔离**：各功能区域独立运行，互不干扰
- **状态实时同步**：任务状态变更实时发布到事件总线
- **双向通信**：主动汇报和被动查询机制

### 3. 自主协调与规划

- **智能路由**：主 Agent 自动拆分分发任务到对应看板区域
- **上下文隔离**：每个子 Agent 专注当前区域任务
- **优先级调度**：支持任务优先级排序

### 4. 周期任务支持

- **多种调度模式**：支持 `at`（一次性）、`every`（周期性）、`cron`（Cron 表达式）
- **主 Agent 统一路由**：Cron 任务通过主 Agent 统一处理，由主 Agent 根据任务性质智能选择通知渠道
- **多渠道通知**：支持单个任务同时通过多个 Channel 发送通知

### 5. 动态模板加载器

- **文件系统加载**：从 Markdown 文件加载 Agent 模板
- **上下文注入**：自动将区域名称、任务上下文、全局配置注入到 Prompt
- **内存缓存**：对加载后的 Agent 配置进行缓存
- **热重载**：支持模板文件变更自动检测并重载

### 6. 多渠道通信支持

支持多种通信渠道接入：

| 渠道 | 状态 | 说明 |
|------|------|------|
| Telegram | ✅ | 支持长轮询、媒体附件、语音转录 |
| Discord | ✅ | 支持消息、提及触发 |
| 钉钉 | ✅ | 流式 SDK 持久连接 |
| 飞书 | ✅ | 事件驱动 Webhook |
| 企业微信 | ✅ | 支持多种机器人模式 |
| Slack | ✅ | Slack Bot API |
| Line | ✅ | Line Bot API |
| Matrix | ✅ | Matrix 协议支持 |
| OneBot | ✅ | QQ/微信机器人协议 |

## 系统架构

```
┌─────────────────────────────────────────────────────────────────┐
│                         主 Agent (Main)                          │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────────────┐  │
│  │   Registry  │  │ Orchestrator │  │   Agent Instances     │  │
│  │  (注册表)    │  │  (协调器)     │  │   (实例管理)          │  │
│  └─────────────┘  └──────────────┘  └───────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      消息总线 (Message Bus)                      │
│                    发布/订阅模式的事件系统                        │
└─────────────────────────────────────────────────────────────────┘
         │                  │                   │
         ▼                  ▼                   ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│   Kanban Board  │ │   Cron Service  │ │   Channels      │

│                 │ │                 │ │                 │
│ ┌─────────────┐ │ │ ┌─────────────┐ │ │ ┌─────────────┐ │
│ │   Zones     │ │ │ │ Job Scheduler│ │ │ │ Telegram    │ │
│ │  (区域管理)  │ │ │ │ (任务调度)   │ │ │ │ HTTP        │ │
│ └─────────────┘ │ │ └─────────────┘ │ │ └─────────────┘ │
│ ┌─────────────┐ │ │                 │ │                 │
│ │   Tasks     │ │ │                 │ │                 │
│ │  (任务管理)  │ │ │                 │ │                 │
│ └─────────────┘ │ │                 │ │                 │
└─────────────────┘ └─────────────────┘ └─────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Agent Workers (执行引擎)                      │
│              每个 Zone 独立的并发任务执行器                        │
└─────────────────────────────────────────────────────────────────┘
```

**关键设计说明：**

1. **Cron 任务流程**：Cron 服务触发的任务会先发送到主 Agent，由主 Agent 根据任务性质决定通过哪些 Channel（单个或多个）进行通知
2. **智能通知路由**：主 Agent 自动分析任务内容，选择合适的通知渠道组合，实现灵活的多渠道通知策略

## 工作流程

```
任务进入 → 看板分区 → 检测无 Agent → 孵化子 Agent → 执行任务 → 完成任务 → 触发重平衡 → 检测无任务 → 释放子 Agent → 汇报状态
```

## 核心模块

| 模块 | 路径 | 功能描述 |
|------|------|----------|
| **看板系统** | `pkg/kanban` | 看板数据结构、任务管理、Agent 编排 |
| **Agent 管理** | `pkg/agent` | Agent 注册表、实例管理、主循环 |
| **周期任务** | `pkg/cron` | 任务调度、定时触发 |
| **消息总线** | `pkg/bus` | 发布/订阅模式的事件系统 |
| **通信渠道** | `pkg/channels` | Telegram、Discord 等多渠道支持 |
| **模板加载器** | `pkg/kanban/loader.go` | 动态模板加载和注入 |

## 部署建议

### 资源配置

| 组件 | CPU | 内存 | 存储 |
|------|-----|------|------|
| 主 Agent | 1 Core | 512MB | 100MB |
| 子 Agent (每个) | 0.5 Core | 256MB | 50MB |
| Kanban | 0.2 Core | 128MB | 200MB |
| Cron | 0.1 Core | 64MB | 50MB |

### 环境变量

```bash
# 看板配置
KANBAN_TEMPLATE_DIR=/path/to/templates
KANBAN_RELOAD_INTERVAL=30s

# Agent 配置
AGENT_MAX_CONCURRENCY=5
AGENT_TIMEOUT=5m

# Cron 配置
CRON_STORE_PATH=/var/lib/octopus/cron.json
```

## 技术栈

- **语言**: Go 1.19+
- **架构风格**: 事件驱动、微内核
- **并发模型**: Goroutine + Channel + Mutex
- **通信协议**: HTTP、WebSocket（计划）、各渠道私有协议

## 架构优势

### 可扩展性

- **水平扩展**: 新增 Zone 无需修改核心代码
- **插件化 Agent**: 支持动态注册不同类型的 Agent
- **多渠道支持**: Channel 接口可轻松扩展新通信方式

### 可靠性

- **原子操作**: 看板状态变更使用锁保护
- **事件溯源**: 所有状态变更都有事件记录
- **优雅降级**: 单个组件故障不影响整体系统

### 可维护性

- **清晰分层**: 数据层、服务层、业务层分离
- **事件驱动**: 组件间低耦合
- **完整日志**: 关键操作都有详细日志记录
